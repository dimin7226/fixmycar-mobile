<a name="introduction"/>

## Введение

Данный документ описывает план тестирования мобильного приложения ["FixMyCar Mobile"](https://github.com/dimin7226/fixmycar-mobile). Документ предназначен для команды тестировщиков, выполняющих тестирование данного проекта. Цель тестирования - проверка соответствия реального поведения программы и ее ожидаемого поведения, которое описано в [требованиях](https://github.com/dimin7226/fixmycar-mobile/blob/main/Test%20plan.md).

<a name="items"/>

## Объект тестирования

В качестве объектов тестирования можно выделить следующие функциональные требования:

* Аутентификация и авторизация пользователей;
* Управление автомобилями клиента (добавление, редактирование, удаление);
* Просмотр списка автомобилей;
* Регистрация нового пользователя;
* Выход из системы.

<a name="quality"/>

## Атрибуты качества

1. Функциональность:
    - функциональная полнота: приложение должно выполнять все заявленные функции;
    - функциональная корректность: приложение должно выполнять все заявленные функции корректно;
2. Удобство использования:
    - все функциональные элементы пользовательского интерфейса имеют названия, описывающие действие, которое произойдет при выборе элемента;
    - навигация между экранами интуитивно понятна;
    - время отклика приложения на действия пользователя не превышает 2 секунд.
3. Безопасность:
    - персональные данные пользователей защищены;
    - авторизация осуществляется через безопасное соединение;
    - приложение корректно обрабатывает неверные учетные данные.
4. Надежность:
    - приложение должно корректно работать при потере и восстановлении соединения;
    - данные должны сохраняться при перезапуске приложения.

<a name="risk"/>

## Риски

В данном случае к рискам можно отнести:
* Нестабильность работы серверной части API;
* Проблемы с сетевой доступностью;
* Несовместимость с некоторыми версиями Android;
* Утечка персональных данных пользователей;
* Некорректная обработка ошибок сервера;
* Проблемы с производительностью при большом количестве автомобилей.

<a name="features"/>

## Аспекты тестирования

В ходе тестирования планируется проверить реализацию основных функций приложения. Аспекты, подвергаемые тестированию: 
* Запуск приложения;  
* Вход в систему;  
* Регистрация нового пользователя;  
* Добавление автомобиля;  
* Редактирование информации об автомобиле;  
* Удаление автомобиля;  
* Просмотр списка автомобилей;  
* Выход из системы;  
* Обработка сетевых ошибок;  
* Валидация вводимых данных;  
* Работа в офлайн-режиме;  
* Восстановление данных после перезапуска приложения.  

### Запуск приложения
Этот вариант использования необходимо протестировать на:
* корректный запуск приложения;
* отображение экрана входа согласно [мокапу](https://github.com/dimin7226/fixmycar-mobile/blob/main/docs/Mockups/StartWindow.png);
* время запуска приложения (не более 3 секунд).

### Вход в систему
Этот вариант использования необходимо протестировать на:
* успешный вход с корректными учетными данными;
* отказ в доступе при неверном номере телефона;
* отказ в доступе при неверном пароле;
* сохранение токена авторизации;
* переход на главный экран после успешного входа;
* обработку случая, когда поля ввода пустые;
* обработку сетевых ошибок во время авторизации.

### Регистрация нового пользователя
Этот вариант использования необходимо протестировать на:
* успешную регистрацию с корректными данными;
* проверку формата номера телефона;
* проверку сложности пароля;
* подтверждение пароля;
* валидацию email адреса;
* обработку случая, когда пользователь с таким номером уже существует;
* переход на экран ввода данных профиля после подтверждения телефона;
* успешное завершение регистрации и автоматический вход;
* обработку сетевых ошибок на каждом этапе регистрации.

### Добавление автомобиля
Этот вариант использования необходимо протестировать на:
* успешное добавление автомобиля с полными данными;
* успешное добавление автомобиля без указания года выпуска;
* проверку обязательных полей (марка, модель, VIN);
* валидацию VIN кода (формат и длина);
* валидацию года выпуска (диапазон от 1900 до текущего года+1);
* отображение нового автомобиля в списке после добавления;
* обработку сетевых ошибок при добавлении;
* сохранение введенных данных при повороте экрана.

### Редактирование информации об автомобиле
Этот вариант использования необходимо протестировать на:
* успешное редактирование информации об автомобиле;
* предзаполнение полей данными автомобиля;
* сохранение изменений;
* отображение обновленных данных в списке автомобилей;
* отмену редактирования без сохранения изменений;
* обработку сетевых ошибок при обновлении данных.

### Удаление автомобиля
Этот вариант использования необходимо протестировать на:
* подтверждение удаления автомобиля;
* успешное удаление автомобиля;
* исчезновение удаленного автомобиля из списка;
* отмену удаления;
* обработку сетевых ошибок при удалении.

### Просмотр списка автомобилей
Этот вариант использования необходимо протестировать на:
* отображение списка автомобилей пользователя;
* отображение информации об автомобиле (марка, модель, год, VIN);
* отображение состояния "Нет автомобилей" при пустом списке;
* обновление списка при добавлении/удалении автомобиля;
* обновление списка при pull-to-refresh;
* отображение детальной информации об автомобиле при нажатии;
* обработку случая с большим количеством автомобилей (прокрутка);
* обработку сетевых ошибок при загрузке списка.

### Выход из системы
Этот вариант использования необходимо протестировать на:
* успешный выход из системы;
* очистку данных авторизации;
* переход на экран входа после выхода;
* невозможность доступа к защищенным экранам после выхода;
* сохранение данных автомобилей в локальном хранилище (при необходимости).

### Обработка сетевых ошибок
Этот вариант использования необходимо протестировать на:
* корректное отображение ошибки при отсутствии интернет-соединения;
* возможность повторить операцию после ошибки;
* сохранение данных при временной потере соединения;
* обработку ошибок сервера (500, 404, 403 и др.);
* отображение понятных сообщений об ошибках пользователю.

### Валидация вводимых данных
Этот вариант использования необходимо протестировать на:
* проверку формата номера телефона;
* проверку длины и сложности пароля;
* проверку формата VIN кода;
* проверку допустимого диапазона года выпуска;
* проверку обязательных полей;
* отображение подсказок при ошибках валидации.

### Работа в офлайн-режиме
Этот вариант использования необходимо протестировать на:
* отображение ранее загруженных данных без интернета;
* невозможность выполнения операций, требующих сеть;
* корректное сообщение об отсутствии соединения;
* автоматическую синхронизацию при восстановлении соединения.

### Восстановление данных после перезапуска приложения
Этот вариант использования необходимо протестировать на:
* сохранение состояния сессии при перезапуске;
* восстановление списка автомобилей после перезапуска;
* сохранение введенных, но не отправленных данных;
* корректную обработку истекшего токена авторизации.

## Нефункциональные требования:
* все функциональные элементы пользовательского интерфейса имеют названия, описывающие действие, которое произойдет при выборе элемента;
* время отклика приложения на действия пользователя не превышает 2 секунд;
* интерфейс адаптирован для различных размеров экранов;
* приложение корректно работает на Android 8.0 и выше;
* персональные данные пользователей хранятся в зашифрованном виде.

<a name="approach"/>

## Подходы к тестированию

При тестировании будет использован ручной подход с использованием следующих методов:

1. **Функциональное тестирование** - проверка соответствия функциональности требованиям;
2. **Тестирование пользовательского интерфейса** - проверка удобства использования и соответствия макетам;
3. **Тестирование безопасности** - проверка защиты данных и корректности авторизации;
4. **Тестирование производительности** - проверка времени отклика и использования ресурсов;
5. **Тестирование совместимости** - проверка работы на разных версиях Android и устройствах;
6. **Тестирование восстановления** - проверка обработки ошибок и восстановления после сбоев.

Тестирование будет проводиться на реальных устройствах и эмуляторах с разными версиями Android.

<a name="conclusion"/>

## Выводы

Данный тестовый план позволяет протестировать основной функционал мобильного приложения FixMyCar Mobile. Успешное прохождение всех тестов не гарантирует полной работоспособности на всех версиях платформ и архитектурах, однако позволяет полагать, что данное программное обеспечение работает корректно в рамках заявленных требований.

Тестирование позволит выявить критические и значительные дефекты на ранних этапах разработки, что снизит стоимость их исправления и повысит качество конечного продукта. Все найденные проблемы будут зафиксированы в системе отслеживания ошибок для последующего исправления разработчиками.
